## Item 3 — Auto-synthesis of `returns_scenarios` warning audit

**Why:** A12's `optimize_fund_portfolio` accepts `returns_scenarios: np.ndarray | None = None`. When None, the code synthesizes a Gaussian scenario matrix from the covariance (multivariate normal, 504 obs, deterministic seed from `fund_ids`). This is intentional back-compat for non-construction callers (block-level pareto, screener drill-downs). The synthesized scenarios lose skewness/kurtosis — silent CVaR misreporting if production paths accidentally trip this.

**Verified state (`optimizer_service.py:616-630`):**

The synthesis already logs `logger.warning("returns_scenarios_missing_using_covariance_synth", n_funds=n)`. That part of Item 3 is DONE (in A12, not deferred). Pending work:

1. **Caller audit** — verify production paths NEVER pass None. Production callers of `optimize_fund_portfolio`:
   - `_run_construction_async` in `model_portfolios.py:2199-2213` — A12 wired `returns_scenarios=sub_returns_scenarios`. **Already correct.**
   - Recursive calls inside `optimize_portfolio` at lines 991, 1072, 1092 — these are BLOCK-level optimizations inside `optimize_pareto_portfolio`. Block IDs, not fund IDs, are passed. Block-level inputs do NOT have raw scenario matrices (blocks aggregate covariance). The synthesis path is the **intended behavior** for these callers. **Already correct — no change needed.**

2. **Telemetry counter** — add a Prometheus-style counter or a dedicated structured log key so a steady-state dashboard can flag any production fund-level run that synthesized scenarios. Implementation:

   In `optimizer_service.py:617-620`, change the existing warning to also include a `caller_kind` tag derived from the call site. Caller passes a hint:

   ```python
   async def optimize_fund_portfolio(
       ...,
       returns_scenarios: np.ndarray | None = None,
       cvar_alpha: float = 0.95,
       caller_kind: str = "unspecified",  # "construction_run" | "screener_preview" | "block_pareto" | "unspecified"
   ) -> FundOptimizationResult:
   ```

   Then in the synthesis branch:
   ```python
   logger.warning(
       "returns_scenarios_missing_using_covariance_synth",
       n_funds=n,
       caller_kind=caller_kind,
   )
   ```

   `_run_construction_async` passes `caller_kind="construction_run"` — production paths should NEVER trigger the warning with this tag. If a log alert sees `caller_kind="construction_run"` paired with the synth warning, it's a P1 bug (real CVaR is not what the operator sees).

   Block-pareto and screener callers pass their own tags (locate them — likely `optimize_pareto_portfolio` and any screener_preview calls).

3. **No hard-fail in production** — DO NOT raise when `returns_scenarios=None` and `caller_kind="construction_run"`. The warning + dashboard is sufficient. Hard-fail can come in a future PR if telemetry shows the warning is silent in production.
