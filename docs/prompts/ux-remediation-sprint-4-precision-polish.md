## What you are fixing

This sprint covers **BL-13 through BL-17** — Structural Consistency items that ensure cross-vertical workflow parity and component consolidation.

### BL-13 — SLA-Aware Loading States

**Problem:** Long-running AI operations (memo generation, DD report generation) use generic spinners with no expected duration indicator. Users cannot distinguish between "processing normally" and "stuck."

**Backlog reference:** BL-13 in `docs/ux/premium-ux-remediation-execution-backlog.md`

**Research needed before implementation:**
- Check `LongRunningAction` component in `@netz/ui` — does it support progress or ETA props?
- Check SSE stream event shapes for memo and DD generation — do they include progress/stage info?
- Identify all long-running operation surfaces in both fronticals

**Acceptance criteria:**
- Long-running operations show stage context ("Extracting evidence...", "Generating chapter 3/8...")
- A subtle progress indicator (not necessarily a bar — could be stage text + elapsed time)
- After exceeding expected SLA (e.g., 3 minutes), show a "Taking longer than expected" message

---

### BL-14 — Streaming Error Recovery

**Problem:** SSE streams for IC memo and DD report generation have no graceful error recovery. If the stream disconnects, the user sees a frozen state with no retry option.

**Research needed before implementation:**
- Read `createSSEStream` utility in `@netz/ui/utils` for retry/error handling
- Check how SSE disconnects are currently handled in ICMemoViewer and DD report pages
- Identify if the backend supports stream reconnection

**Acceptance criteria:**
- SSE disconnection shows an inline error with "Retry" button
- Retry resubscribes to the same job stream (not restart the operation)
- Partial content received before error is preserved (not cleared)

---

### BL-15 — Conviction Confidence Calibration

**Problem:** DD reports include `confidence_score` and `decision_anchor` but these are not visually calibrated. A confidence of 0.72 vs 0.95 looks the same. No visual gradient or threshold indicators.

**Research needed before implementation:**
- Check how `confidence_score` and `decision_anchor` are used in DD report pages
- Check if there are existing confidence visualization patterns in Credit

**Acceptance criteria:**
- Confidence score is displayed with visual calibration (color gradient from warning to success)
- Threshold bands are visible (e.g., <0.6 = low/warning, 0.6-0.8 = moderate/info, >0.8 = high/success)
- Decision anchor text is displayed alongside the confidence score

---

### BL-16 — Negative Screening Emphasis

**Problem:** The wealth screener has eliminatory criteria (Layer 1) that should be visually distinct from soft scoring criteria. Rejected funds should be clearly marked as eliminated, not just filtered.

**Research needed before implementation:**
- Read screener page in Wealth frontend
- Check screener API response shape for eliminatory vs. scoring results
- Identify current rendering of screening results

**Acceptance criteria:**
- Eliminatory rejections are visually distinct (danger/red treatment, struck-through or dimmed)
- Reason for elimination is displayed inline
- Passed eliminatory filters show a subtle "Passed" indicator

---

### BL-17 — Override Audit Narrative

**Problem:** When a user overrides an AI recommendation (e.g., approving despite low confidence, rejecting despite high confidence), no narrative is captured explaining the override rationale beyond the standard approval/rejection rationale.

**Research needed before implementation:**
- Check if existing approval/rejection flows detect when the action contradicts AI recommendation
- Check if override-specific fields exist in the API

**Acceptance criteria:**
- When a user action contradicts the AI recommendation, the confirmation dialog adds an "Override Rationale" section
- Override rationale is recorded in the audit trail
- Override events are visually flagged in the audit trail panel
